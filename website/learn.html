01

The big idea

Your data tool runs
on your machine.

Most ETL tools are heavy: a cluster, a control plane, a bill for compute you barely used. Duckle is the opposite - a desktop studio with DuckDB running in-process. The whole engine lives inside the app, so nothing leaves your laptop and there is no server to wait on.

  • The canvas is the program. Boxes wired together - one reads, one reshapes, one writes. You watch the rows flow through.
  • It compiles to SQL you can read. Every diagram becomes ordinary DuckDB SQL you can open, copy, and run yourself.
  • One file, one binary. A pipeline is a single .json. Bundle it into a standalone executable and drop it on any server.

07

Connections & secrets

Save it once.
Never hardcoded.

Save a connection and node fields fill themselves from it - change a credential in one place and everything follows. Secrets and your git token are encrypted at rest with AES-256-GCM using a per-workspace key.

In exported pipelines and over MCP, secrets stay as ${ENV:KEY} placeholders. Paths take runtime tokens too - ${workspace}, ${date}, ${timestamp} - so one pipeline handles many days and folders.

08

The engine · the magic

Your graph becomes a plan of stages.

When you press Run, the engine sorts your nodes in order and turns each into a stage - almost always a CREATE OR REPLACE VIEW. String them together and you have the whole program.

  • Views by default. Lazy, so a later filter pushes down and only needed rows are read. Reused stages become real tables automatically.
  • External databases get ATTACHed as duckle_src - the same SELECT whether you preview or run.
  • One session when it can. A pure-SQL pipeline folds into a single DuckDB run; driver, HTTP, AI and code stages run on their own.
  • Reject ports route failing rows to a separate relation, lazily, in one pass.
Why is a stage a lazy view, not a copied table, by default?
Exactly. A lazy view lets a later WHERE or projection push into the source scan - nothing is copied, and only the needed rows are read. The engine materializes a table only when a stage is reused enough that copying pays off.

-- 1. read the source as a lazy view
CREATE OR REPLACE VIEW "orders_csv" AS
  SELECT * FROM read_csv_auto('orders.csv', header=true);

-- 2. the filter is just a WHERE
CREATE OR REPLACE VIEW "keep_paid" AS
  SELECT * FROM "orders_csv" WHERE status = 'paid';

-- 3. the sink is a COPY
COPY (SELECT * FROM "keep_paid")
  TO 'out/paid.parquet' (FORMAT parquet);

Ship it

Four ways to put it to work.

  • Run and schedule. On demand, on an interval, or a full cron expression evaluated in your local time.
  • Headless runner and web console. duckle-runner runs a file with no GUI; duckle serve hosts Operations, Pipelines, Runs and the scheduler.
  • Build a single binary. Bundle the engine and your pipeline into one self-contained executable, secrets by env or passphrase.
  • Let an LLM build it. The MCP server connects Claude in one click to list, create, validate, run and build - real, runnable pipelines.

And Duckie, the local Qwen assistant, does the same on device - describe a pipeline and it streams one onto the canvas, prompts and data never leaving your machine. In-app git commits and pushes your workspace and watches CI.

14

Trust & the frontier

Know exactly what changed.

  • Time travel and data diff. Read a table as of a past snapshot on DuckLake, and diff two snapshots to see what changed between two points in time.
  • Column lineage and impact. Trace any output column back to its sources, and see what breaks if you change it - before you change it.
  • Signed runs, drift and a trust score. Every run signs into a reproducible .ducklock; schema-drift detection and a trust score flag when a source's shape moved.
  • Contracts and data branches. Pin a source's expected shape, and test a change against real data on a branch before it touches the main copy.
